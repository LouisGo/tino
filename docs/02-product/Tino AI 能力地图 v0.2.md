# Tino AI 能力地图 v0.2

> 日期：2026-04-06
> 适用阶段：剪贴板模块 + 应用底座已落地，AI 主链路启动前
> 本版变化：从 `review-first` 修正为 `result-first`，明确 AI 是后台知识编译器，审阅是隐藏干预层
> MVP 心智：`inbox-first`。用户不需要先创建 `topic`，系统会在后台从混杂输入中生成并收敛议题。

---

## 1. 一个总判断

`Tino` 的 AI 不应先被理解为前台聊天 Agent，也不应先被理解为批次审阅产品。
它首先应被理解为：

> 一个把个人信息流静默编译成 Markdown 知识资产的后台知识编译器。

更贴近用户语言的说法是：

> 一个 AI 驱动的私人信息收件箱。你先把东西丢进来，系统再替你长出议题。

对普通用户来说，真正重要的是：

- 信息能低摩擦进入系统
- 用户不需要在输入前思考它属于什么议题
- 系统能在后台把碎片整理成结果
- 结果能稳定落到 `topics/` / `_inbox/`
- 只有在异常时，用户才需要干预

---

## 2. AI 在 Tino 中的四个环节

### 2.1 Capture

- 剪贴板只是第一入口，不是唯一入口
- 后续可以扩展到导入、链接转录、IM 记录等来源
- 关键不是来源多少，而是统一进入同一条 capture 主链路

### 2.2 Compile

- AI 在后台做切分、归并、聚类、命名、摘要、置信度判断
- 这一层是 harness 约束下的编译器，不是自由发挥的通用 Agent
- 它的目标是稳定、可控、可审计，不是前台表演“思考过程”

### 2.3 Deliver

- 用户真正感知价值的地方是结果层，而不是处理中间层
- 高置信结果应进入系统生成的 `topics/`
- 低置信、冲突或暂时不适合落盘的内容应进入 `_inbox/`

### 2.4 Intervene

- 审阅、调试、纠偏仍然需要存在
- 但它们应是隐藏的次级入口
- 主要服务：
  - 开发期校准
  - 异常兜底
  - 高级用户修正

`/ai` 这类界面可以存在，但不应成为普通用户的主舞台。

---

## 3. 短期聚焦

短期只做一条真实闭环：

> `静默输入 -> 后台归并与理解 -> 系统生成议题 -> 结果落盘 -> 异常可干预`

### 3.1 结构化批次输出

AI 对批次至少输出：

```text
source_ids / decision / topic_slug_suggestion / topic_name_suggestion / title / summary / key_points / tags / confidence / reason
```

`decision` 枚举保持：

- `archive_to_topic`
- `send_to_inbox`
- `discard`

### 3.2 批次内聚类

AI 不应逐条孤立处理碎片，而应识别：

- 哪些内容属于同一件事
- 哪些内容应作为一个 section 一起落盘
- 哪些内容更适合先进入 `_inbox`

### 3.3 结果交付优先

用户优先看到的应是：

- 系统最近长出了哪些议题
- 哪些内容被并入了已有议题
- 哪些结果进入 `_inbox`
- 哪些异常需要处理

而不是持续暴露完整的 AI 中间过程。

### 3.4 隐藏干预层

`/ai` 或类似面板应主要承担：

- 开发期校准
- 低置信度内容处理
- 错误流向修正
- 输出质量抽检

它不应被当成“用户每天都要打开的核心体验”。

### 3.5 受控上下文

AI 处理批次时，只允许看到：

- 当前批次
- Top N 相关 topic 索引摘要
- 必要的用户偏好

不要无限喂历史全文。

---

## 4. 中期方向

在短期闭环跑通之后，再推进：

- 用户偏好学习
- topic 生命周期管理
- 周期性回顾
- 历史重编译
- 可复用素材层

这些能力的前提不是“审阅面板更复杂”，而是“知识出口已经稳定”。

---

## 5. 长期方向

长期可以承认“有记忆的陪伴层”存在价值。
但前提是：

- capture 足够真实
- compile 足够稳定
- deliver 足够可靠

否则所谓陪伴型 Agent 只会退化成普通聊天壳。

---

## 6. 当前阶段不该做什么

- 把 `/ai` review 做成主产品舞台
- 把聊天窗口当成第一入口
- 做通用工具调用 Agent
- 做复杂主动提醒系统
- 做大规模长期记忆注入

---

## 7. 阶段性成功标准

短期 AI 主链路跑通的标志不是“用户愿不愿意逐条 Accept”，而是：

- 用户在输入前不需要思考议题结构
- 用户大多数时候不需要逐批操作
- `topics/` 输出质量稳定、结构可复用
- `_inbox` 只承接少量真正需要关注的异常
- 用户知道系统在持续替他清理信息债，而不是制造新的整理负担
