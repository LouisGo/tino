# 环境与打包流程

> 最后更新：2026-04-10

这份文档只回答三件事：

- 开发环境怎么跑
- 测试 / staging 包怎么打、怎么装
- 正式 production 包怎么打、怎么装

## 1. 环境矩阵

| 环境 | 入口命令 | `TINO_APP_ENV` | 数据通道 | macOS App Identity | 用途 |
| --- | --- | --- | --- | --- | --- |
| Development | `pnpm tauri:dev` | `development` | `shared` | `com.louistation.tino.development` | 日常开发、UI 调试、命令联调 |
| Test / Staging | `pnpm tauri:build-install:test` | `staging` | `shared` | `com.louistation.tino.preview` | 测试真实打包行为、权限、剪贴板回填 |
| Production | `pnpm tauri:build-install:prod` | `production` | `production` | `com.louistation.tino.production` | 真正给用户使用的正式包 |

## 2. 开发环境

开发环境使用：

```bash
pnpm tauri:dev
```

说明：

- 这是未打包的开发运行态。
- 数据仍写入 `shared` 通道，方便和 preview/test 共享测试数据。
- macOS 剪贴板回填、Accessibility、签名相关问题，不要在这个环境里做最终判断。
- 原因是 `dev` 运行的是开发态进程，不是最终安装到 `/Applications` 的打包 app。

适合做的事：

- UI 调整
- IPC 联调
- Rust / Renderer 基础行为验证
- 非权限类功能开发

不适合做的事：

- 验证 macOS Accessibility 是否稳定
- 验证回填到外部应用的真实行为
- 验证系统权限是否会持久保存

## 3. Test / Staging 环境

测试环境必须使用打包后的 preview app。

推荐命令：

```bash
pnpm tauri:build-install:test
```

等价命令：

```bash
pnpm tauri:build-install:staging
```

它会完成：

1. 生成 `staging` / `preview` 构建
2. 对 `.app` 做 post-process 签名校验
3. 直接把验证通过的 app 安装到 `/Applications/Tino Preview.app`

如果只想重新安装、而不重新构建：

```bash
pnpm tauri:install:test
```

### 3.1 为什么不能手动从 DMG 拖进去

这轮问题已经证明，手动走 DMG 拖拽链路时，开发期很容易把错误 bundle 或不稳定签名副本装进 `/Applications`。

因此当前约定是：

- 开发和测试阶段默认不要手动从 DMG 拖拽安装
- 默认用 `pnpm tauri:build-install:test`
- 安装脚本会对源 bundle 和目标 `/Applications` 副本都做 `codesign` 校验

### 3.2 macOS 本地签名

如果要稳定验证：

- Clipboard paste-back
- Accessibility 授权
- 回填到 Cursor / 微信 / Arc / Ghostty 等外部应用

则必须先准备本地签名身份：

```bash
pnpm macos:setup-local-signing
```

说明：

- 该脚本会生成本地 code-signing 证书与 keychain 配置
- 构建脚本会优先读取 `.local/macos-signing.env`，其次读取 `~/.config/tino/macos-signing.env`
- 如果没有本地签名身份，构建会退回 ad-hoc signing
- ad-hoc signing 可以跑，但 macOS Accessibility 授权可能会在 rebuild 后失效
- 如果配置里写着某个签名身份，但该 identity 已经失效，打包会直接失败并提示重新执行 `pnpm macos:setup-local-signing`

## 4. Production 环境

正式包使用：

```bash
pnpm tauri:build-install:prod
```

它会：

1. 切换到 `production` app env
2. 使用 `production` 数据通道
3. 生成 `Tino.app`
4. 安装到 `/Applications/Tino.app`

如果你只执行：

```bash
pnpm tauri:build:prod
```

那它只会产出正式包，不会安装到 `/Applications`。

当前行为：

- build 成功后会在 Finder 中直接揭示真正可安装的源 bundle：
  `src-tauri/target/release/bundle/macos/Tino.app`
- 这个 `Tino.app` 才是当前团队默认的安装源
- 如果只是双击 `.dmg` 并直接从挂载卷里运行 app，那是在临时卷中启动，不等于已经安装到 `/Applications`
- 因此 Finder 的“应用程序”里没有 `Tino` 是正常的
- 真正安装请用 `pnpm tauri:build-install:prod`，或者手动把 `Tino.app` 复制到 `/Applications/Tino.app`

正式环境与测试环境隔离点：

- 产品名不同
- Bundle identifier 不同
- 应用数据目录不同
- 日志目录不同
- 知识根数据通道不同

当前正式环境标识：

- `productName`: `Tino`
- `identifier`: `com.louistation.tino.production`

正式对外分发时，优先使用团队正式签名身份。`pnpm macos:setup-local-signing` 生成的是本机开发签名，只适合本地开发和测试，不适合作为面向最终用户的签名方案。

## 5. 构建产物

`pnpm tauri build` / `pnpm tauri build --production` 之后，macOS 侧会有三类产物：

- `.app`
- `.dmg`
- `.zip`

当前团队默认把 `.app` 作为安装源：

- `.app` 是 post-process 后立即验证过的源 bundle
- `.zip` 是给需要绕过 DMG 链路时的备用分发产物
- `.dmg` 仍然保留，但开发期不作为默认安装路径；如果当前机器的 `hdiutil` 不可用，构建会保留 `.app/.zip` 并跳过 `.dmg`
- 从 `.dmg` 里直接运行只是在挂载卷里启动，不是安装行为

## 6. 当前团队约定

日常开发：

```bash
pnpm tauri:dev
```

测试剪贴板回填 / 权限 / 真机行为：

```bash
pnpm macos:setup-local-signing
pnpm tauri:build-install:test
```

发正式包：

```bash
pnpm tauri:build-install:prod
```

如果只是本地先看构建产物：

```bash
pnpm tauri:build:prod
```

构建完成后 Finder 会自动揭示 `Tino.app`，避免误去点 DMG。

本地签名坏掉时：

```bash
pnpm macos:setup-local-signing
```

## 7. 不要做的事

- 不要用 `pnpm exec tauri ...` 绕过 wrapper 脚本
- 不要在 dev 环境里判断 Accessibility 是否真的可用
- 不要默认手工从 DMG 拖到 `/Applications`
- 不要让 `development`、`preview`、`production` 共享同一个 macOS bundle identifier
