# 个人信息流软件 MVP 技术冻结记录

> 日期：2026-04-06
> 目的：冻结 MVP 第一阶段的关键实现边界，避免项目骨架和主链路开发阶段反复返工

## 1. 冻结原则

- 只冻结会影响目录结构、模块职责、运行模型的关键决策
- 低优先级实现细节采用默认方案，不再继续发散
- 本文优先级高于临时讨论结论，后续如需调整，应显式更新本文

## 2. 用户已明确拍板的关键决策

### 2.1 应用形态

- 应用需要 `macOS 常驻后台`
- 需要 `menubar / tray + Dock + 主窗口`
- 首次启动默认直接进入 `主窗口`
- 关闭主窗口时只 `hide`，不退出应用
- 点击 `Dock` 图标打开主窗口
- 点击 `menubar` 图标也打开主窗口
- MVP 需要支持 `开机自启动`
- 开机自启动默认开启，但应发生在 `首次初始化完成之后`

### 2.2 主窗口职责

MVP 主窗口只承载以下五类能力：

- 当前状态概览
- 知识根目录设置
- AI Provider 设置
- 最近结果 / 异常入口 / 队列调试信息
- 手动触发批处理、暂停 / 恢复采集

补充边界：

- 如存在 `/ai` 或批次审阅界面，默认只作为隐藏或次级干预入口
- 不把批次审阅设计成普通用户的日常主流程

明确不进入 MVP 的内容：

- 聊天窗口
- 笔记编辑器
- 历史内容浏览器
- 可视化统计面板
- 以审阅面板为核心的日常操作流

### 2.3 知识根目录

- 首次启动时让用户选择知识根目录
- 默认建议路径为 `~/tino-inbox`

### 2.4 剪贴板采集边界

- MVP 采集 `纯文本 + 富文本`
- 非文本类型方向上需要保留 `图片 / 视频 / 文件` 的占位和回显能力
- 但第一阶段主链路只真正处理 `纯文本 + 富文本`
- 完全相同内容的重复复制默认去重
- 启动前历史剪贴板：有低成本现成能力就接，没有就不做 MVP

### 2.5 系统层职责分布

- `剪贴板轮询` 放在 `Rust` 侧
- `本地文件读写` 统一放在 `Rust` 侧
- `AI 调用层` 先放在前端
- `Agent runtime / prompt 编排 / streaming UI` 放在 `Renderer`
- `Agent tools 的真实副作用执行` 继续走 `Rust command`
- `_system/` 运行态先用 `JSON` 文件存储，不上数据库

### 2.6 AI SDK 与 Provider 方案

- 前端 AI 调用使用 `Vercel AI SDK Core`
- Provider 适配层使用 `@ai-sdk/openai-compatible`
- `Vercel AI SDK` 只负责 `Renderer` 侧的模型调用与 Agent 编排，不直接接管本地文件系统
- Provider 配置统一为：
  - `baseURL`
  - `apiKey`
  - `model`
- MVP 先做结构化批处理，不做聊天流式 UI
- AI 配置支持两种来源：
  - 主窗口写入本地配置文件
  - `.env` 作为回退来源
- 优先级：`本地配置文件 > .env`

### 2.7 配置分层

- `知识配置` 与 `应用敏感配置` 分开存储
- 知识相关规则放知识根目录下的 `_config/`
- AI Provider、API Key、运行态等敏感配置放应用自己的 `app data`

### 2.8 批处理触发策略

- AI 处理采用 `混合触发`
- 阈值为：
  - 满 `20` 条立即处理
  - 或最晚 `10` 分钟处理一次

### 2.9 崩溃恢复策略

- 未处理缓冲区在异常退出、崩溃、系统重启后
- 下次启动时 `自动恢复并继续处理`

### 2.10 去重与过滤

- 去重策略：`完全相同内容`
- 去重窗口：`5 分钟`
- MVP 接受“硬编码最小过滤”
- 被过滤内容：
  - 不进入 `daily`
  - 不进入 AI
  - `_system` 保留过滤日志

### 2.11 归档与输出策略

- `daily/` 采用 `按天一个 Markdown 文件`
- `topics/` 采用 `每个 topic 一个长期 Markdown 文件`
- 每次 AI 处理结果作为一个新 `section` 追加到对应 topic 文件
- `_inbox/` 采用 `按天一个 Markdown 文件`

### 2.12 AI 分类与 topic 规则

- 允许自动创建新 `topic`
- 新 `topic` 必须满足 `高置信度`
- 低置信度内容一律进入 `_inbox`
- AI 批次内的内容允许拆分到多个 topic
- 批次内输出粒度采用 `先聚类再输出`
- 默认目标是静默把高置信结果直接送入 `topics/`
- 审阅 / 调试界面只作为异常兜底、质量校准和高级修正入口

### 2.13 Topic 内容结构

- `topics` 中每个 section 固定包含：
  - 标题
  - 摘要
  - 关键点列表
  - 原始片段引用
  - 元数据：时间、tags、source_ids、confidence

### 2.14 Daily 原始归档策略

- `daily` 只做原始归档，不做 AI 重写
- 富文本同时保留：
  - 可读版本
  - 原始表示或可回溯表示

### 2.15 前端运行模型

- 主窗口关闭后只做 `hide`，不销毁
- 前端 runtime 保持存活
- 后台 AI 批处理继续依赖隐藏状态下的主窗口运行

### 2.16 未配置 AI 时的行为

- 知识根目录配置完成后，即可开始监听与 `daily` 归档
- AI Provider 未配置时，不进入 AI 批处理
- 这段期间的内容：
  - 只进入 `daily`
  - 不进入正式 AI 待处理队列
- 用户完成 AI 配置后：
  - 不自动补跑历史内容
  - 仅支持用户手动补跑

### 2.17 历史补跑策略

- 历史补跑按 `天` 为粒度
- 即选择某一天的 `daily` 文件进行补跑
- 当前缓冲区处理与历史补跑是两个独立操作

### 2.18 暂停语义

- `Clipboard Watch` 与 `Capture Pipeline` 分离
- 剪贴板监听能力默认始终存在
- `暂停采集` 的语义是：停止进入系统主链路
- 暂停采集后，新复制内容：
  - 只被监测
  - 不写 `daily`
  - 不进缓冲区
  - 不进 AI

### 2.19 批次失败与并发

- AI 批次失败后自动重试 `3` 次
- 超过 `3` 次后标记失败并 `pass`
- 不阻塞后续新批次
- `schema` 校验失败也纳入同一套失败重试机制
- AI 批次按批次 `串行执行`
- 历史补跑与实时批次共用同一个执行队列

### 2.20 AI 上下文范围

- AI 上下文采用：
  - `当前批次`
  - `受控的已有 topic 上下文`
- 严禁把完整 `topics/*.md` 无限喂给模型
- 每次只给模型有限数量的相关 topic 索引

### 2.21 Topic 索引维护

- 每次某个 topic 文件有新 section 追加后
- 顺手刷新该 topic 的摘要索引

## 3. 为实现稳定性补充的默认方案

以下内容未逐项向用户继续追问，直接作为默认实现冻结。

### 3.1 内容类型枚举

统一保留 `content_kind` 字段，至少支持：

- `plain_text`
- `rich_text`
- `image`
- `video`
- `file`

### 3.2 CaptureRecord 基础字段

建议最小字段如下：

- `id`
- `source`
- `captured_at`
- `content_kind`
- `raw_text`
- `raw_rich`
- `hash`

### 3.3 Topic 索引传给模型的内容

每次只传有限的 `topic index`，而不是全文。单条索引建议包含：

- `topic_name`
- `topic_slug`
- `topic_summary`
- `recent_tags`
- `last_updated_at`

默认只带 `Top 5` 个最相关 topic。

### 3.4 Topic 路径控制

- AI 只输出 topic 建议
- 最终 `topic slug`、文件名、路径全部由程序生成

### 3.5 `_system/` 默认文件职责

建议包含：

- `queue.json`
- `runtime.json`
- `filters.log`
- `batches/`
- `logs/`

### 3.6 AI 输出原则

- AI 输出必须先结构化
- 程序负责 schema 校验
- 程序负责最终 Markdown 渲染与文件写入

## 4. 仍未冻结但不阻塞开发的事项

以下事项可以在项目骨架阶段或模块实现阶段再定，不阻塞现在开工：

- 具体 Markdown section 模板长什么样
- 富文本的原始表示采用 HTML、RTF 还是其他格式
- topic 相似度判断的具体算法
- AI 配置 UI 的字段校验细节
- app data 的最终目录结构
- 首次初始化向导的具体交互

## 5. 对下一步开发的直接影响

基于当前冻结结果，下一步可以直接进入 `项目骨架搭建`，重点是：

1. 初始化 `Tauri + React + TypeScript`
2. 建立 `Rust` 侧的：
   - 剪贴板轮询
   - 文件读写
   - `_system` 状态管理
3. 建立前端侧的：
   - AI SDK 调用层
   - 主窗口设置页
   - 最近批次与运行状态调试面板
4. 搭出主链路空壳：
   - Capture
   - Orchestrator
   - Archive
   - AI Pipeline
   - Knowledge Output

MVP 第一阶段已经具备足够稳定的技术边界，不需要再继续发散式讨论。
