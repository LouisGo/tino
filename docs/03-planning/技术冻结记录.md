# 个人信息流软件 MVP 技术冻结记录

> 日期：2026-04-13
> 目的：冻结 MVP 当前仍有效的关键实现边界，避免 AI rethink 之后继续沿旧 `/ai review` 方案漂移

## 1. 冻结原则

- 只冻结会影响目录结构、模块职责、运行模型的关键决策
- 不再把旧 `/ai review` 过渡方案当成未来 AI 主链路
- 本文优先级高于临时讨论结论，后续如需调整，应显式更新本文

## 2. 用户已明确拍板的关键决策

### 2.0 MVP 用户心智

- MVP 采用 `inbox-first`，不是 `topic-first`
- 用户不需要先创建 `topic`
- 用户不需要先决定内容该归到哪里
- `topic` 是系统在后台基于混杂输入逐步归并、命名和收敛出来的结果
- `topics/` 是当前知识出口，不代表用户必须先理解或管理该结构

### 2.1 应用形态

- 应用需要 `macOS 常驻后台`
- 需要 `menubar / tray + Dock + 主窗口`
- 首次启动默认直接进入 `主窗口`
- 关闭主窗口时只 `hide`，不退出应用
- 点击 `Dock` 图标打开主窗口
- 点击 `menubar` 图标也打开主窗口
- MVP 需要支持 `开机自启动`

### 2.2 主窗口职责

MVP 主窗口只承载以下能力：

- 当前状态概览
- 知识根目录设置
- AI Provider / capability 设置
- 最近结果 / AI runtime 状态 / 异常入口 / 队列调试信息
- 手动触发批处理、暂停 / 恢复采集

补充边界：

- 如存在 AI Ops、批次检查页或 legacy `/ai` 页面，默认只作为隐藏或次级入口
- 不把手动审阅面板设计成普通用户的日常主流程

明确不进入 MVP 的内容：

- 聊天窗口作为主入口
- 以审阅面板为核心的日常操作流
- 复杂可视化统计面板
- 让用户预先维护 topic 结构的工作流

### 2.3 知识根目录

- 首次启动时让用户选择知识根目录
- 默认建议路径为 `~/tino-inbox`

### 2.4 剪贴板采集边界

- MVP 采集 `纯文本 + 富文本`
- 非文本方向保留 `图片 / 视频 / 文件` 的占位和回显能力
- 第一阶段主链路仍以 `纯文本 + 富文本` 为核心
- 完全相同内容的重复复制默认去重
- 启动前历史剪贴板：有低成本现成能力就接，没有就不做 MVP

### 2.5 系统层职责分布

- `剪贴板轮询` 放在 `Rust`
- `本地文件读写` 放在 `Rust`
- `交互式 AI 调用` 放在 `Renderer`
- `后台 AI 批处理 runtime` 放在 `Rust async runtime`
- `真实副作用执行` 继续走 `Rust`
- `_system/` 运行态与审计短中期继续允许用 `JSON`
- `反馈记忆 / 质量指标 / 偏好信号` 放本地 `SQLite`

补充边界：

- 后台 AI 编译不能依赖 React 组件生命周期
- 不再冻结“后台 AI 批处理继续依赖隐藏主窗口 renderer 存活”这条旧假设
- provider settings 保留，但不是 AI 模块的架构中心

### 2.5.1 Renderer <-> Rust IPC 范式

- 新增或重构的 renderer IPC API，统一按 `Query` / `State Command` / `Action Command` / `Subscription` 分类
- `Query` 只读快照，不承载副作用；优先使用 `async command`
- `State Command` 只表达用户意图；Rust 维护权威状态机，提交成功后由 Rust 主动广播 typed event
- `Action Command` 保留 request/response 语义，用于窗口控制、系统打开、paste-back 等即时副作用
- `Subscription` 用于持续同步；事件不能替代冷启动或 remount 时的快照查询
- 重查询、同步文件 IO、sqlite、JSON 处理、图片处理等阻塞工作，不应停留在 Tauri 主线程

### 2.6 AI capability 与 provider 方案

- Provider 设置页保留，用于内部或调试阶段输入 `baseURL` / `apiKey` / `model`
- Provider profile 继续采用“多 profile + 当前启用项”
- Provider settings 是能力来源之一，不是业务模块的架构中心
- 交互式 AI 可以继续使用 `Vercel AI SDK Core + @ai-sdk/openai`
- 后台编译必须通过 Rust-owned capability boundary 执行，不依赖 React 生命周期
- mock / 注入式 capability source 允许存在，用于开发和调试
- 业务模块不得直接依赖某个页面里的 provider UI 状态
- 敏感配置优先级仍为：`本地配置文件 > .env`

### 2.7 配置与存储分层

- `知识配置` 与 `应用敏感配置` 分开存储
- 知识相关规则放知识根目录下的 `_config/`
- provider profile、API Key、当前启用项、运行态等敏感配置放应用稳定持久化目录
- feedback memory、quality metrics、偏好 SQLite 也放应用稳定持久化目录
- clipboard history 在 retention 窗口内属于长期用户资产，不应因应用升级、覆盖安装或 bundle identifier 变更而丢失
- macOS 下应用侧持久化默认落到 `~/Library/Application Support/Tino/{shared|production}`

### 2.8 批处理触发策略

- AI 处理采用 `混合触发`
- 触发常量由 Rust 后台 runtime 持有，不做前端临时 UI 配置
- MVP 当前默认值仍为：
  - 满 `20` 条立即处理
  - 或最晚 `10` 分钟处理一次

### 2.9 崩溃恢复策略

- 未处理缓冲区在异常退出、崩溃、系统重启后
- 下次启动时 `自动恢复并继续处理`

### 2.10 过滤与去重

- 第一层去重策略：`完全相同内容`
- 去重窗口：`5 分钟`
- MVP 接受“硬编码最小过滤”
- 最小过滤至少覆盖：
  - 空文本
  - 超短文本
  - 疑似密码 / token / 验证码
- 被过滤内容：
  - 不进入 `daily`
  - 不进入 AI
  - `_system` 保留过滤日志

补充边界：

- 近重复或句子级相似度筛查允许作为 LLM 前置 gate
- 但必须保守，避免把应保留的信息误杀

### 2.11 归档与输出策略

- clipboard 插件缓存与知识资产必须分层处理
- clipboard history 不等同于长期知识真相源，但在 retention 窗口内必须按 durable asset 对待
- `daily/` 采用 `按天一个 Markdown 文件`
- `daily/` 只做原始归档，不做 AI 重写
- `topics/` 采用 `每个系统生成 topic 一个长期 Markdown 文件`
- 每次 AI 处理结果作为一个新 `section` 追加到对应 topic 文件
- `_inbox/` 采用 `按天一个 Markdown 文件`
- AI 不直接控制真实文件路径

### 2.12 AI 分类与 topic 规则

- `topic` 在 MVP 中是系统生成的分组结果，不是用户预先创建的容器
- 允许自动创建新 `topic`
- 新 `topic` 必须满足 `高置信度`
- 低置信度内容进入 `_inbox`
- 批次内输出粒度采用 `先聚类再输出`
- 默认目标是静默把高置信结果直接送入 `topics/`
- 用户主路径不是 review-first
- legacy `/ai` 或类似人工审阅面不是 AI 模块的主舞台

### 2.13 Topic 内容结构

- 当前持久化策略下，`topics` 中每个 section 固定包含：
  - 标题
  - 摘要
  - 关键点列表
  - 原始片段引用
  - 元数据：时间、tags、source_ids、confidence

### 2.14 前端运行模型

- 主窗口关闭后只做 `hide`
- 前端 runtime 可继续承载交互式状态
- 但后台 AI 批处理不能依赖隐藏主窗口 renderer 存活
- 复杂 renderer 交互面的键盘语义不应依赖浏览器默认 `Tab` 焦点流

### 2.15 未配置 AI capability 时的行为

- 知识根目录配置完成后，即可开始监听与 `daily` 归档
- 未配置可用的后台 AI capability 时，不进入后台 AI 批处理
- 这段期间的内容：
  - 只进入 `daily`
  - 不进入正式 AI 待处理队列
- 用户补齐能力配置后：
  - 不自动补跑历史内容
  - 仅支持用户手动补跑

### 2.16 历史补跑策略

- 历史补跑按 `天` 为粒度
- 即选择某一天的 `daily` 文件进行补跑
- 当前缓冲区处理与历史补跑是两个独立操作

### 2.17 暂停语义

- `Clipboard Watch` 与 `Capture Pipeline` 分离
- 剪贴板监听能力默认始终存在
- `暂停采集` 的语义是：停止进入系统主链路
- 暂停采集后，新复制内容：
  - 只被监测
  - 不写 `daily`
  - 不进缓冲区
  - 不进 AI

### 2.18 批次失败与并发

- AI 批次失败后自动重试 `3` 次
- 超过 `3` 次后标记失败并 `pass`
- 不阻塞后续新批次
- `schema` 校验失败也纳入同一套失败重试机制
- AI 批次按批次 `串行执行`
- 历史补跑与实时批次共用同一个执行队列

### 2.19 AI 上下文范围

- AI 上下文采用：
  - `当前批次`
  - `受控的已有 topic 上下文`
  - `必要的用户偏好记忆`
- 严禁把完整 `topics/*.md` 无限喂给模型
- 每次只给模型有限数量的相关 topic 索引

### 2.20 反馈与质量指标

- `用户主动纠错率` 是 AI 主链路第一指标
- 当前目标：`< 5%`
- 至少记录三类本地信号：
  - 分类纠错 / reroute
  - topic 使用与访问模式
  - 删除与保留行为
- 这些信号不做成前端草稿态，而应作为 Rust 可消费的本地持久化资产

### 2.21 Topic 索引维护

- topic index 需要成为正式资产，而不是长期依赖临时扫目录推导
- 每次某个 topic 文件有新 section 追加后，应顺手刷新该 topic 的摘要索引

## 3. 冻结后的直接含义

从现在开始：

- 不再继续围绕旧 `/ai review` 方案扩 AI UI
- 不再把 provider 设置页当成 AI 架构中心
- 不再把后台编译放在 renderer 生命周期里理解
- 后续 AI 模块工作先做 contract、状态机、存储、反馈，再做 AI Ops UI
