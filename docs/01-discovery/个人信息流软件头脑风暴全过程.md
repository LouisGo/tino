## 关键设计决策说明

​**Layer 2 的 Capture Node 是核心**，它的职责只有一件事：把所有输入源的内容统一成同一个数据结构，然后进队列。不做分类，不做判断，只做收口。这样 Layer 3 的 AI 决策可以独立迭代，不跟输入源耦合。

**Layer 3 分两个并行节点**是刻意的。AI 分类决策是默认路径，但 Skill/Config 是用户的覆盖层。你说的"以后可以写 Skill 控制 AI 把特定内容存到特定位置"，就是这里实现的。Config 是 JSON，Skill 是 prompt snippet，两者都在运行时被注入到 AI 决策的 prompt 里。

​**初始目录结构建议如下**，足够抽象但有形：

```
~/knowledge-inbox/
  _inbox/          ← 未分类暂存，AI 处理前的缓冲
  topics/          ← AI 自动创建和命名的主题目录
  daily/           ← 按日期的流水记录（降级方案）
  _config/         ← config.json + skills/
```

AI 默认行为：判断内容属于哪个 topic，如果 topic 不存在就创建，如果置信度不够就扔进 `_inbox`。用户随时可以通过 config 覆盖。

## 剪贴板问题的结论

**macOS 上不存在剪贴板变化的 push 通知。**  这是系统级限制，Raycast、Maccy 等所有剪贴板工具的实现方式完全一样：轮询 `NSPasteboard.changeCount`，这个计数器在每次剪贴板内容变化时自增，只要比较前后两次的值是否不同，就能知道内容是否变化。

0.5 秒的轮询间隔 CPU 占用低于 1%，这是业界惯例，性能完全不是问题。

**对你的影响：**  Tauri 的 Rust 后端起一个 0.5s 的定时器轮询 `NSPasteboard.changeCount`，检测到变化就把内容推到前端队列。这个逻辑只有十几行，没有任何技术风险。

这个盲点已经消除。进入评审。

## 评审：3 轮关键问题

**Q1：Capture Node 的触发边界**

剪贴板每次变化都自动进队列，还是需要用户做一个"确认动作"才进？

两种模式差别很大：自动模式你什么都不用做，但会有大量噪音（密码、临时复制的内容）；确认模式需要多一个快捷键动作，但进来的内容质量高很多。Raycast 是自动的，但它有黑名单过滤。你倾向于哪种？

---

**Q2：AI 决策的触发时机**

内容进来之后，AI 分类是实时触发（每条内容进来立刻调用一次 API），还是批量触发（比如每隔 5 分钟或攒够 10 条再统一处理）？
